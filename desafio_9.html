Desafio 9

a)

O package.json é usado por todos os projetos no espaço de trabalho, incluindo o projeto de aplicativo inicial que é criado pela CLI ao criar o espaço de trabalho.

Inicialmente, este package.json inclui um conjunto inicial de pacotes, alguns dos quais são exigidos pelo Angular e outros que suportam cenários de aplicativos comuns. Você adiciona pacotes ao package.json à medida que seu aplicativo evolui. Você pode até remover alguns.

O package.json está organizado em dois grupos de pacotes:

  • As dependências que são essenciais para a execução de aplicativos.
  • DevDependencies que são necessários apenas para desenvolver aplicativos.
    A seção de dependências do package.json contém:
  • Angular packages: Núcleo Angular e módulos opcionais; seus nomes de pacotes começam @angular/.
  • Support packages: bibliotecas de terceiros que devem estar presentes para que os aplicativos Angular sejam executados.
  • Polyfill packages: Polyfills preenchem lacunas na implementação JavaScript de um navegador.

b)

O angular.json é um arquivo no nível raiz do workspace Angular, fornece padrões de configuração específicos do projeto e de todo o workspace para ferramentas de construção e desenvolvimento fornecidas pela CLI do Angular. Os valores de caminho fornecidos na configuração são relativos à pasta root do workspace.

c)

O router outlet atua como um espaço reservado que o Angular preenche dinamicamente com base no estado atual do roteador.

Um router outlet emite um evento de ativação quando um novo componente é instanciado, um evento de desativação quando um componente é destruído. Um evento anexado é emitido quando o RouteReuseStrategy instrui a saída a reconectar a subárvore, e o evento desanexado é emitido quando o RouteReuseStrategy instrui a saída a desanexar a subárvore.

d)

A Declaração 1 é eager-load carrega os dados mesmo que você não vá utilizá-los.

A Declaração 2 é lazy-load e será carregada 'on demand' (apenas quando você solicitar explicitamente o carregamento desta).

e)

Os componentes definem visualizações, que são conjuntos de elementos de tela que o Angular pode escolher e modificar de acordo com a lógica e os dados do seu programa.

Os componentes usam serviços, que fornecem funcionalidades específicas não diretamente relacionadas às visualizações. Os provedores de serviços podem ser injetados em componentes como dependências, tornando seu código modular, reutilizável e eficiente.

Você define a lógica do aplicativo de um componente — o que ele faz para dar suporte à exibição — dentro de uma classe. A classe interage com a view por meio de uma API de propriedades e métodos. Angular cria, atualiza e destrói componentes à medida que o usuário se move pelo aplicativo. Seu aplicativo pode agir a cada momento neste ciclo de vida por meio de ganchos de ciclo de vida opcionais.

f)

Foi criado o base-resource-form, onde todas as informações e configurações de formulario fora inseridas em um unico arquivo para que os outros componentes herdem essas configurações. Foi criado o base-resource-list onde ficou toda a parte logica das paginas que listam as categorias e os lançamentos. Assim também o compoente de migalha de pao, o cabecalho, foi criado um componente para os erros dos campos do formulario, os erros de servidor e por fim a barra de navegação.

g)

@Input() permite que um componente pai atualize dados no componente filho. Neste caso, cria uma lista dentro do objeto BreadCrumbItem;

h)

@Output() permite que o filho envie dados para um componente pai, o Output é um decorator que marca um campo como uma propriedade de saida, a propriedade da DOM vinculada a esta saida é atualizada automaticamente quando ocorre alterações.

i)

  • Core é o módulo projetado para os serviços Singleton compartilhados em todo o aplicativo. (funcionamento da aplicação)
  • Pages é o módulo onde colocamos os componentes responsáveis pela view das páginas do projeto. (renderizado na tela)
  • Shared é o módulo que trata da funcionalidade compartilhada, é onde coloca-se os componentes, pipes e diretivas reutilizáveis. (compartilhamento de dados e funções)

j)

O arquivo in-memory-database serve para criar um "back-end" mockado no projeto.

k)

    Métodos:
  • setValue(): define o valor de um controle do formGroup.
  • clearValidators(): esvazia a lista de validadores síncronos.
  • get(): obtém o valor de um controle ou caminho.
  • getError(): informa dados de erros ao controle conforme o caminho.
  • patchValue(): corrige o valor do FormGroup, recebe um objeto e faz o possível para corresponder os valores ao controle correto no Formgrup.
    Propriedades:
  • invalid: define um controle como inválido de acordo com o seu status
  • validator: retorna a função usada para determinar se o formulario esta valido de forma síncrona.
  • disabled: define o status do controle para desabilitado.
  • value: exibe os valores dos campos no formulario.
  • root: recupera o ancestral de nível superior do controle.
